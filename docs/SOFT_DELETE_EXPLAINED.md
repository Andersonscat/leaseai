# 🔄 Soft Delete - Как это работает

## ❓ Вопрос: Удаляется ли property из базы данных?

## ✅ Ответ: НЕТ! Property остается в базе данных

---

## 🎯 Что происходит при удалении

### В UI:
1. Вы нажимаете "Delete Property"
2. Подтверждаете удаление
3. Property **исчезает из списка**
4. Показывается: "🗑️ Property deleted successfully!"

### В базе данных:
```sql
-- НЕ делается Hard Delete:
-- DELETE FROM properties WHERE id = '123';  ❌

-- Делается Soft Delete:
UPDATE properties 
SET deleted_at = '2026-01-23 10:30:15'
WHERE id = '123';  ✅
```

**Property все еще в таблице!** Просто помечен как удаленный.

## 🔍 Как увидеть удаленные?

### Вариант 1: Supabase Dashboard

1. Откройте Supabase Dashboard
2. **Table Editor** → `properties`
3. Найдите колонку `deleted_at`
4. Увидите: NULL (активные) и timestamp (удаленные)

### Вариант 2: SQL запрос

Запустите в SQL Editor:

```sql
-- Файл: supabase/check-deleted-properties.sql
SELECT 
  address,
  price,
  CASE 
    WHEN deleted_at IS NULL THEN '✅ Active'
    ELSE '🗑️ Deleted'
  END as status,
  deleted_at
FROM properties
ORDER BY deleted_at DESC NULLS LAST;
```

**Результат:**
```
┌─────────────┬────────┬──────────┬─────────────────┐
│ address     │ price  │ status   │ deleted_at      │
├─────────────┼────────┼──────────┼─────────────────┤
│ 456 Pike St │ $2850  │ 🗑️ Deleted│ 2026-01-23 10:30│
│ 123 Main St │ $2500  │ ✅ Active │ NULL            │
└─────────────┴────────┴──────────┴─────────────────┘
```

## 🔓 Как восстановить удаленное

### Через SQL (Supabase):

```sql
-- Восстановить property
UPDATE properties 
SET deleted_at = NULL 
WHERE id = 'ваш-property-id';
```

**Готово!** Property снова появится в списке! 🎉

## 💡 Зачем Soft Delete?

### Преимущества:

1. **Случайно удалили?**
   - ✅ Можно восстановить!
   - ❌ Hard delete = навсегда потеряно

2. **История сохраняется**
   - Контракты остаются
   - Сообщения сохраняются
   - Связи не нарушаются

3. **Аудит**
   ```sql
   -- Кто и когда удалил?
   SELECT address, deleted_at 
   FROM properties 
   WHERE deleted_at IS NOT NULL;
   ```

4. **Безопасность**
   - Нет риска потерять важные данные
   - Можно отследить изменения
   - Легко откатить ошибку

## ⚖️ Сравнение методов

| Что происходит? | Hard Delete | Soft Delete (наш) |
|-----------------|-------------|-------------------|
| Property в БД? | ❌ Удален | ✅ Остается |
| Видно в UI? | ❌ Нет | ❌ Нет (скрыт) |
| Можно восстановить? | ❌ Нет | ✅ Да |
| Контракты сохраняются? | ⚠️ Удаляются (CASCADE) | ✅ Сохраняются |
| История? | ❌ Потеряна | ✅ Сохранена |
| Размер БД? | 📉 Не растет | 📈 Растет медленно |

## 🗑️ Постоянное удаление (опционально)

Если нужно освободить место в БД:

```sql
-- Удалить НАВСЕГДА properties старше 30 дней
-- ⚠️ ВНИМАНИЕ: Необратимо!
DELETE FROM properties 
WHERE deleted_at IS NOT NULL 
  AND deleted_at < NOW() - INTERVAL '30 days';
```

Это можно настроить как:
- Cron job (каждую ночь)
- Supabase Function (автоматически)
- Ручная кнопка "Очистить архив"

## 📋 FAQ

**Q: Почему не видно удаленные в UI?**  
A: API фильтрует их: `.is('deleted_at', null)`

**Q: Сколько места занимают удаленные?**  
A: Немного. Можно чистить раз в месяц.

**Q: Можно ли вернуть удаленное?**  
A: Да! Через SQL или будущую вкладку Archive.

**Q: Что если удалить property с контрактами?**  
A: Контракты сохранятся! (property_id останется)

**Q: Нужно ли чистить удаленные?**  
A: Не обязательно, но можно раз в 30-90 дней.

**Q: Можно ли увидеть кто удалил?**  
A: Сейчас нет, но можно добавить поле `deleted_by`.
